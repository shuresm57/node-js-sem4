ES Modules and Static Files

ES Modules vs CommonJS

Enabling ES Modules

If we add "type": "module" to our package.json, it enables ES Modules syntax in Node.js. 

{
  "type": "module",
  "dependencies": {
    "express": "5.2.1"
  }
}

CommonJS Exporting

Exporting:

const cookie = require('./cookie.json');

function getCookie() {
  return cookie;
}

module.exports = { getCookie };

CommonJS Importing

Importing:

const cookiesUtil =
  require('./util/commonjsCookiesUtil.js');

This is the old way and we will not do that anymore. Instead we will use ES Modules.

ES Modules Exporting

Exporting:

// Named export
export function esModuleCookieFactory() {
  return 'On a break :D';
}

// Default export
export default {
  esModuleCookieFactory
};

ES Modules Importing

Importing:

// Named import (curly braces,
// can define more than one)
import { esModuleCookieFactory }
  from './util/esModuleCookiesUtil.js';

// Default import (no curly braces,
// for single imports)
import cookiesUtil
  from './util/esModuleCookiesUtil.js';

Always use the .js extension in ES Module imports.

Traditional Script Loading

Regular scripts share global scope — but order matters and can be hard to manage:

<!-- Scripts share global scope -->
<script src="./common.js"></script>
<script src="./cookieFactory.js"></script>

Module Script Loading

With type="module", every script has its own scope and can share code. The browser handles load order automatically:

<!-- ES Module - isolated scope -->
<script type="module"
  src="./cookies.js">
</script>

Named vs Default Exports

Named exports (recommended):

// Export multiple
export function func1() { }
export const value = 42;

// Import specific
import { func1, value }
  from './module.js';

Default export:

// One per file
export default function() { }

// Import without braces
import myFunction from './module.js';

Static File Management

Express Static Middleware

Without static file serving, users can't access CSS, JS or images. We need to give explicit permission to Express in order to serve these files. 

import express from 'express';
const app = express();

// Serve files from 'public' directory
app.use(express.static('public'));

Absolute Path Requirements

Always use path.resolve — it creates absolute paths, which are required by sendFile. Note that __dirname is CommonJS-specific and won't work with ES Modules.

import path from 'path';

app.get('/', (req, res) => {
  res.sendFile(path.resolve(
    'public/frontpage/frontpage.html'
  ));
});

app.get('/cookieFactory', (req, res) => {
  res.sendFile(path.resolve(
    'public/cookieFactory/cookieFactory.html'
  ));
});

Typical Project Structure

project/
├── public/           # client
│   ├── frontpage/
│   │   ├── frontpage.html
│   │   └── frontpage.css
│   └── cookieFactory/
├── util/             # server
│   └── esModuleCookiesUtil.js
├── package.json      # server
└── app.js            # server

HTTP Redirection Techniques

JavaScript Redirection

Timed redirect with JavaScript:

setTimeout(() => {
  window.location.href = '/success';
}, 2000);

HTML Meta Redirection

Or with a meta tag:

<meta http-equiv="refresh"
  content="3;url=/success">

Useful for redirecting users after a form submission or similar action.

Express Server Redirection

app.get('/old-page', (req, res) => {
  res.redirect('/new-page');
});

Summary

  • Use ES Modules with "type": "module" — never use CommonJS
  • Always include .js in ES Module imports
  • Use app.use(express.static('public')) for static files
  • Use path.resolve() for file paths
  • Client redirection: JavaScript or meta tags (user actions)
  • Server redirection: res.redirect()
  • ES Modules work in both browsers and Node.js
